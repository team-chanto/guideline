1HTML — 要素の使い分け

見た目ではなく「その要素が何を意味するか」で選ぶ。
以下の要素の使い分けが社内の基準となる。

1-1. ページ構造を作る要素

要素いつ使うかよくある間違い
headerページ全体、または各セクションの導入部分（ロゴ・ナビ・見出しなど）ページ上部にあるからといって何でも header に入れる。中身に導入的な役割がなければ不要
mainページ固有のメインコンテンツ。1ページに1つだけmain が複数ある、または header/footer を main の中に入れてしまう
footerページ全体、または各セクションの補足情報（コピーライト・関連リンクなど）フッターっぽい見た目だから footer にする。意味的に「補足・末尾情報」かで判断する
nav主要なナビゲーションリンクのまとまり（グローバルナビ・パンくず等）ページ内のあらゆるリンクの塊に nav をつける。主要な導線にだけ使う
section見出し（h2 等）を持つ、意味的なまとまり見出しを持たない section。スタイル目的なら div を使う
articleそれ単体で独立して成立するコンテンツ（ブログ記事・ニュース・カード1枚など）section との混同。「その部分だけ切り出して別の場所に貼っても意味が通じるか？」で判断する
div意味を持たない汎用コンテナ。スタイルやレイアウトの都合で囲みたいときに使う意味のある要素（section、article、nav 等）の代わりに使う

1-2. テキスト・コンテンツ要素

要素いつ使うかよくある間違い
h1〜h6見出し。h1 → h2 → h3 の順で、階層を飛ばさないフォントサイズを変えたいだけで h3 を使う。見出し階層は文書構造で決める
p段落（ひとまとまりの文章）余白を作りたいだけで空の p を置く。余白は CSS で制御する
spanインライン要素の汎用コンテナ。テキストの一部にスタイルを当てたいときに使うブロック要素の代わりに使う。span はインライン要素なので、レイアウト用途には div を使う
br同じ段落内での改行（住所・詩の行分け等）レイアウト目的の改行。余白調整やブロック間の改行には使わない
ul順序に意味がないリスト（ナビ項目・特徴一覧など）リストなのに div を並べる
ol順序に意味があるリスト（手順・ランキングなど）手順なのに ul で書く
dl / dt / dd用語とその説明のペア（FAQ・スペック表・定義リストなど）単なるラベルと値の並びに ul > li を使っている。対応関係がある場合は dl が適切
table行と列で構造化されるデータ（料金表・比較表・スケジュール等）レイアウト目的で table を使う。表形式のデータにだけ使う

div と span の使い分け：div = ブロック要素の汎用コンテナ（レイアウト・囲み用）、span = インライン要素の汎用コンテナ（テキストの一部にスタイルを当てる用）。どちらも意味を持たないので、意味のある要素で代替できる場合はそちらを使う。

1-3. インタラクティブ要素

要素いつ使うかよくある間違い
a別ページ・別URLへの遷移。href 属性が必須href="#" にして JS でクリックイベントを処理する → それは button の役割
buttonその場で何かを実行する操作（モーダルを開く、タブを切り替える等）見た目がボタンだから a にスタイルを当てる。遷移しないなら button を使う
判断基準：「クリックしたらどこかに飛ぶか？」→ Yes なら a、No なら button。

1-4. 画像要素

要素いつ使うかよくある間違い
imgコンテンツとして意味のある画像。alt 属性は必須（後述）装飾画像に img を使って意味のない alt を書く → 装飾は CSS background-image で
pictureデバイスや画面幅に応じて出し分ける画像。source で条件を指定し、img をフォールバックにするすべての画像を picture にする必要はない。出し分け不要なら img だけで十分

picture の基本構文：

<!-- picture + source でフォーマット・デバイスの出し分け -->
<picture>
  <source srcset="/img/hero-sp.webp" media="(max-width: 767px)" type="image/webp">
  <source srcset="/img/hero-pc.webp" type="image/webp">
  <img src="/img/hero-pc.jpg" alt="メインビジュアルの説明" width="1280" height="600">
</picture>

width / height 属性は必ずつける

img タグには width と height を必ず指定する。

<!-- ✅ OK -->
<img src="/img/photo.jpg" alt="写真の説明" width="800" height="600">

<!-- ❌ NG — width / height がない -->
<img src="/img/photo.jpg" alt="写真の説明">

理由：ブラウザが画像の読み込み前にスペースを確保できるようになり、読み込み中にレイアウトがガタッとずれる現象（CLS: Cumulative Layout Shift）を防げる。指定しないとページの表示品質と Core Web Vitals のスコアに悪影響が出る。

値はカンプや画像の実寸から取得する。CSS で width: 100% にして可変にする場合でも、HTML 側の属性はアスペクト比の算出に使われるので省略しない。

loading 属性と fetchpriority 属性

画像の読み込みタイミングを制御する属性。ページの表示速度に直結するので、すべての img に意識的に設定する。

属性値意味いつ使うか
loadinglazy画面外にある画像は、スクロールで近づくまで読み込まないファーストビュー以外の画像すべて
loadingeager（デフォルト）ページ読み込みと同時に取得するファーストビュー内の画像
fetchpriorityhighブラウザに「この画像を優先的に取得して」と伝えるファーストビューのメインビジュアル（LCP 要素）
fetchprioritylow優先度を下げるページ下部のあまり重要でない画像
 
<!-- ✅ ファーストビューのメインビジュアル（LCP 候補） -->
<img
  src="/img/hero-main.webp"
  alt="メインビジュアルの説明"
  width="1280" height="600"
  fetchpriority="high"
>

<!-- ✅ ファーストビュー内だが、メインビジュアルではない -->
<img
  src="/img/header-logo.svg"
  alt="サイトロゴ"
  width="200" height="40"
>

<!-- ✅ ファーストビュー外（スクロールしないと見えない） -->
<img
  src="/img/about-team-photo.webp"
  alt="チーム集合写真"
  width="800" height="600"
  loading="lazy"
>
判断基準：ページを開いた瞬間に見えるか見えないかで分ける。
画像の位置loadingfetchpriority
ファーストビュー内 — メインビジュアルeager（デフォルト、省略可）high
ファーストビュー内 — それ以外eager（デフォルト、省略可）指定なし
ファーストビュー外lazy指定なし（必要に応じて low）
よくある間違いなぜダメか
すべての画像に loading="lazy" をつけるファーストビューの画像まで遅延されると、ユーザーに見える画像の表示が遅くなる。LCP スコアも悪化する
fetchpriority="high" を大量につける全部が高優先だとブラウザが優先順位をつけられない。メインビジュアル等の1〜2枚に限定する
loading="lazy" と fetchpriority="high" を同時につける矛盾する指定。遅延したいのか優先したいのかブラウザが判断できない

画像フォーマット

フォーマット用途備考
WebP写真・イラスト全般。基本はこれを使うJPEG/PNG より軽量。ほぼ全ブラウザ対応済み
JPEGWebP 非対応環境のフォールバック、または納品指定がある場合—
PNG透過が必要な画像ファイルサイズが大きくなりがちなので、透過不要なら WebP を使う
SVGアイコン・ロゴ等のベクター画像拡大しても劣化しない。色変更も CSS で可能

基本方針：写真系は WebP、アイコン・ロゴ系は SVG。picture + source で WebP を優先しつつ、img の src にフォールバック画像（JPEG/PNG）を指定する。

1-5. id 属性の使い方

用途可否理由
アンカーリンクの遷移先OKページ内リンクの遷移先として id を使うのは正当な用途
label の for 属性OKフォーム要素との紐付けは id が必要
CSS のスタイル指定使わない詳細度が高く再利用できない。class を使う
DOM 操作のフック（JS）使わないdata-* 属性 + querySelector で取得する（4章参照）
アニメーションのフック使わないトリガーや対象要素の指定には data-* 属性を使う
計測タグ（GA イベント等）先方指定がない限り使わないdata-* 属性で対応。id は一意制約があり管理が煩雑
 
<!-- ✅ OK — アンカーリンクの遷移先は id -->
<section id="about">...</section>
<a href="#about">会社概要へ</a>

<!-- ✅ OK — フォームの紐付けは id -->
<label for="email">メールアドレス</label>
<input id="email" type="email">

<!-- ❌ NG — 計測タグに id を使う（先方指定がない場合） -->
<button id="cta-button">お問い合わせ</button>

<!-- ✅ OK — 計測タグは data 属性で管理 -->
<button data-gtm="cta-button">お問い合わせ</button>

2HTML — 属性・メタ情報

2-1. alt 属性

基本ルール

画像の種類alt の書き方例
意味のある画像画像の内容を端的に説明するalt="代表取締役 山田太郎のプロフィール写真"
装飾画像空の alt を設定するalt=""
リンク内の画像（テキストなし）リンク先の情報を書くalt="会社概要ページへ"
図・グラフ凡例や数値をもれなく書く（後述）—
必須alt 未設定（属性自体がない）は NG。空でもいいから必ず alt 属性をつける。

alt の書き方：「画像を説明する」とは

alt は 「この画像が表示されなかったときに、代わりに何が伝わればいいか」 を書く。判断基準は 人が読み上げたときに自然かどうか。

<!-- ✅ OK — 画像の内容が伝わる -->
<img src="/img/office.webp" alt="渋谷オフィスの受付エリア。白を基調とした内装に木製カウンターがある" width="800" height="600">

<!-- ❌ NG — ファイル名をそのまま書いている -->
<img src="/img/office.webp" alt="office.webp" width="800" height="600">

<!-- ❌ NG — 「画像」「写真」から始める（読み上げ時に「画像: 画像の〜」と冗長になる） -->
<img src="/img/office.webp" alt="画像：オフィスの写真" width="800" height="600">

<!-- ❌ NG — 過不足がある -->
<img src="/img/office.webp" alt="写真" width="800" height="600">
やりがちなミスなぜダメか正しい対応
ファイル名をそのまま書く内容が伝わらない画像に写っているものを説明する
「画像」「写真」「イメージ」から始めるスクリーンリーダーが「画像:」を自動で読み上げるため二重になる内容から直接書き始める
全部同じ alt にするページ内で複数の画像が同じ説明では区別がつかない画像ごとに固有の説明を書く
過剰に長い alt読み上げが冗長になる1〜2文で端的に。長くても50文字程度を目安にする

図・グラフの alt

図やグラフの場合は、視覚的に読み取れる情報（凡例・数値・傾向）をもれなく書く。見えない人が alt だけで内容を把握できる状態を目指す。

<!-- ✅ OK — 凡例と数値をもれなく書く -->
<img
  src="/img/sales-chart.webp"
  alt="2025年度の四半期別売上グラフ。Q1: 1,200万円、Q2: 1,500万円、Q3: 980万円、Q4: 2,100万円。Q4に最も売上が伸びている"
  width="600" height="400"
>

<!-- ✅ OK — 比較表の場合 -->
<img
  src="/img/plan-comparison.webp"
  alt="料金プラン比較表。ライトプラン: 月額980円、ストレージ10GB。スタンダードプラン: 月額1,980円、ストレージ50GB。プロプラン: 月額4,980円、ストレージ無制限"
  width="800" height="400"
>

<!-- ❌ NG — 凡例や数値がない -->
<img src="/img/sales-chart.webp" alt="売上グラフ" width="600" height="400">

<!-- ❌ NG — 「グラフ」だけ -->
<img src="/img/sales-chart.webp" alt="グラフ" width="600" height="400">

図・グラフの alt を書くときのポイント：

  • 何のグラフかを最初に書く（「2025年度の四半期別売上グラフ」）
  • 凡例をもれなく書く（項目名と数値のセット）
  • 傾向や結論があれば最後に添える（「Q4に最も売上が伸びている」）
  • 読み上げたときに自然な日本語になるか を声に出して確認する

2-2. ページごとに必須のメタ情報

すべてのページで以下が入っていることを確認する。

項目確認すること
<title>ページごとに固有のタイトルが入っているか。全ページ同じになっていないか
<meta name="description">ページの内容を要約した説明文が入っているか（120〜160文字目安）
OGP 情報og:title / og:description / og:image / og:url / og:type が設定されているか

OGP の基本セット：

<meta property="og:title" content="ページタイトル">
<meta property="og:description" content="ページの説明文">
<meta property="og:image" content="https://example.com/ogp.jpg">
<meta property="og:url" content="https://example.com/page/">
<meta property="og:type" content="website">
<meta name="twitter:card" content="summary_large_image">

確認方法：OGP の表示確認は SNS のデバッガー（Facebook Sharing Debugger 等）で実際の表示を見る。

3CSS

3-1. クラス命名規則

命名規則は スタイルのスコープが自動で閉じるかどうか で使い分ける。

環境命名規則理由
WordPress / 素の HTML + SCSS など（module 化できない）BEMCSS がグローバルスコープのため、命名規則で衝突を防ぐ必要がある
Astro / Next.js など（module 化できる）シンプルな命名でOKフレームワークがスコープを自動で閉じるため、BEM の冗長さが不要

module 化できない環境 → BEM

概念命名ルール例
Block独立したコンポーネント.card
ElementBlock の中の構成要素。__ で繋ぐ.card__title
Modifier状態やバリエーション。-- で繋ぐ.card--featured
NGなぜダメかOK
.card__title__iconElement を多段ネストしない。BEM は2階層まで.card__icon
.cardTitleBEM はケバブケース＋__/-- で区切る.card__title
.red-button見た目ベースの命名.button--primary

module 化できる環境 → シンプルな命名

Astro の Scoped CSS や Next.js の CSS Modules ではクラス名がハッシュ化されるため、BEM を使わなくてもクラス名の衝突は起きない。

/* Card.module.css（Next.js）/ <style> 内（Astro） */
.card { ... }
.title { ... }        /* BEM なら .card__title だが、スコープが閉じているので不要 */
.featured { ... }     /* BEM なら .card--featured */

ただし、以下のルールは環境を問わず共通：

  • 見た目ベースの命名は避ける（.red-button → .primary）
  • 役割が伝わる名前にする（.wrapper1 → .hero-section）
  • ケバブケースで統一する（.cardTitle → .card-title）

3-2. SCSS のルール

スタイルの「影響範囲」を理解する

CSS は本来 グローバルスコープ で動く。つまり、あるファイルに .card { color: red; } と書いたら、サイト内のすべての .card に影響する。これが CSS で最も事故が起きやすいポイント。

この「影響範囲の制御」には、大きく分けて2つのアプローチがある。

① フレームワークが自動でスコープを切る方法（CSS Modules）

Astro や Next.js では、コンポーネントごとにスタイルのスコープが自動で閉じる仕組みがある。

// Astro の場合 — <style> に書いたスタイルはそのコンポーネント内だけに効く
// Component.astro
<div class="card">...</div>
<style>
  .card { color: red; }  /* このファイルの .card にしか効かない */
</style>

// Next.js (CSS Modules) の場合 — ファイル名を .module.css にする
// Card.module.css
.card { color: red; }  /* import したコンポーネント内の .card にしか効かない */

// Card.jsx
import styles from './Card.module.css';
<div className={styles.card}>...</div>

これらの仕組みでは、ビルド時にクラス名が .card_x1a2b のようにハッシュ付きの一意な名前に変換される。だから他のコンポーネントの .card と衝突しない。フレームワークが影響範囲を自動で管理してくれる。

② 自分でセレクタを使ってスコープを切る方法（SCSS + BEM）

CSS Modules のような仕組みがない環境（WordPress のテーマ、素の HTML + SCSS のプロジェクトなど）では、自分でクラス名の設計によって影響範囲を制御する必要がある。これが BEM とページクラスによるネストの役割。

// ✅ .p-about で囲むことで「会社概要ページ内の .card」に影響範囲を限定
.p-about {
  .card { color: red; }       /* TOPページの .card には影響しない */
  .card__title { ... }
}

// ✅ .p-top で囲むことで「TOPページ内の .card」に限定
.p-top {
  .card { color: blue; }      /* 会社概要ページの .card には影響しない */
  .card__title { ... }
}
<!-- about.html -->
<main class="p-about">
  <div class="card">...</div>  <!-- color: red が当たる -->
</main>

<!-- top.html -->
<main class="p-top">
  <div class="card">...</div>  <!-- color: blue が当たる -->
</main>
環境影響範囲の制御方法誰がスコープを管理するか
Astro / Next.js（CSS Modules）フレームワークがクラス名をハッシュ化フレームワークが自動で管理
WordPress / 素の HTML + SCSSページクラス + BEM で命名を設計自分で管理する
重要どちらのアプローチでも「このスタイルはどこに影響するのか」を常に意識すること。SCSS で何気なく書いた .card { ... } が、意図しないページの .card を壊す事故は実務で頻繁に起きる。

ネスト禁止（ページ区切り以外）

SCSS のネストは ページクラスによるスコープの区切り としてのみ使う。それ以外のネストは禁止。

// ✅ OK — ページ単位の区切りとしてのネスト
.p-about {
  .card { ... }
  .card__title { ... }
}

// ❌ NG — BEM をネストで再現
.card {
  &__title { ... }
  &--active { ... }
}

& によるクラス名の省略も禁止

& を使って BEM のクラス名を組み立てない。理由は2つ。

  1. 検索性が下がる：.card__title でファイル内を検索しても、&__title ではヒットしない
  2. 生成される CSS が見えづらい：ネストが深くなるとどんなセレクタが出力されるか追いにくい
// ❌ NG — & でクラス名を省略
.card {
  &__title { color: red; }
  &__body { font-size: 14px; }
  &--featured { background: yellow; }
}

// ✅ OK — クラス名をそのまま書く
.card__title { color: red; }
.card__body { font-size: 14px; }
.card--featured { background: yellow; }

3-3. 論理プロパティ

margin-inline: auto 以外の論理プロパティは使わない。

今後の予定CSS Lint で物理プロパティ／論理プロパティの統一ルールを整備する予定。現時点ではルールが未確定のため、混在を避ける。
/* ✅ OK */
.container { margin-inline: auto; }
.card { margin-top: 16px; padding-left: 24px; }

/* ❌ NG — margin-inline: auto 以外の論理プロパティ */
.card { margin-block-start: 16px; padding-inline-start: 24px; }

3-4. hover の実装：any-hover を使う

hover スタイルは @media (any-hover: hover) で囲む。タッチデバイスで hover が残り続ける問題を防ぐため。

SCSS / 素の CSS の場合

/* ✅ OK */
@media (any-hover: hover) {
  .button:hover {
    opacity: 0.7;
  }
}

/* ❌ NG — メディアクエリなしで hover を書く */
.button:hover {
  opacity: 0.7;
}

4-1. DOM の取得：data 属性 + querySelector

DOM を操作する要素は data-* 属性をつけて、querySelector で取得する。class や id で DOM を取得しない。

<!-- ✅ OK — data 属性をつける -->
<button data-action="toggle-menu">メニュー</button>
<nav data-element="menu">...</nav>
// ✅ OK — querySelector + data 属性
const trigger = document.querySelector('[data-action="toggle-menu"]');
const menu = document.querySelector('[data-element="menu"]');

trigger.addEventListener('click', () => {
  menu.classList.toggle('is-open');
});
// ❌ NG — class で DOM を引っ張る
const trigger = document.querySelector('.js-menu-trigger');
// → class はスタイル用。JS のフックと CSS が密結合になる

// ❌ NG — getElementById
const menu = document.getElementById('menu');
// → id はフォームの label 紐付け以外で使わない方針

理由：class 名の変更で JS が壊れるのを防ぐ。data-* 属性なら「この要素は JS から参照される」ことが HTML を見ただけでわかる。

4-4. 関数化：変数操作が連続したら分割する

変数の操作が連続で続く場合は、処理のまとまりごとに関数に切り出す。「何をしているか」が関数名で伝わる状態を目指す。

// ❌ NG — 変数操作がベタ書きで連続し、何をしているか追いにくい
const items = document.querySelectorAll('[data-element="item"]');
const prices = [];
items.forEach((item) => {
  const raw = item.dataset.price;
  const num = parseInt(raw, 10);
  if (!isNaN(num)) { prices.push(num); }
});
const total = prices.reduce((sum, p) => sum + p, 0);
const tax = Math.floor(total * 0.1);
const totalWithTax = total + tax;
const formatted = totalWithTax.toLocaleString();
document.querySelector('[data-element="total"]').textContent = `¥${formatted}`;
// ✅ OK — 処理のまとまりを関数に切り出す
function collectPrices() {
  const items = document.querySelectorAll('[data-element="item"]');
  const prices = [];
  items.forEach((item) => {
    const num = parseInt(item.dataset.price, 10);
    if (!isNaN(num)) prices.push(num);
  });
  return prices;
}

function calcTotalWithTax(prices) {
  const total = prices.reduce((sum, p) => sum + p, 0);
  const tax = Math.floor(total * 0.1);
  return total + tax;
}

function renderTotal(amount) {
  const el = document.querySelector('[data-element="total"]');
  if (!el) return;
  el.textContent = `¥${amount.toLocaleString()}`;
}

// メインの流れが一目でわかる
const prices = collectPrices();
const total = calcTotalWithTax(prices);
renderTotal(total);
判断基準：「この処理を1行の日本語で説明できるか？」を考える。説明できるまとまりが関数1つの粒度。

5-2. aria-label / aria 属性

aria-label は スクリーンリーダーに「この要素が何か」を伝える ための属性。見た目ではわからない情報を補完する。

いつ使うか

状況対応例

  <svg><!-- ハンバーガーアイコン --></svg>

<!-- ✅ OK — 「詳しく見る」が複数ある場合、aria-label で区別 -->

<!-- ❌ NG — 複数の「詳しく見る」が区別できない -->
<a href="/news/1">詳しく見る</a>
<a href="/news/2">詳しく見る</a>

aria-hidden — 装飾要素を読み上げから除外する

意味を持たない装飾目的の画像・HTML要素には aria-hidden="true" を設定し、スクリーンリーダーの読み上げ対象から除外する。

装飾画像・アイコン:

<!-- ✅ OK — テキストの隣にある装飾アイコンは aria-hidden で除外 -->
<a href="/contact">
  <svg aria-hidden="true"><!-- 矢印アイコン --></svg>
  お問い合わせ
</a>

<!-- ✅ OK — テキストと併記される装飾画像 -->
<div class="card">
  <img src="/img/decoration-line.svg" alt="" aria-hidden="true">
  <p>サービスの説明文...</p>
</div>

<!-- ❌ NG — 装飾アイコンが読み上げられてしまう -->
<a href="/contact">
  <svg><!-- 矢印アイコン --></svg>
  お問い合わせ
</a>

装飾的な HTML 要素:

<!-- ✅ OK — 見た目だけの区切り線やバッジ装飾は aria-hidden -->
<span class="badge" aria-hidden="true">NEW</span>
<h3>新サービスのご案内</h3>

<!-- ✅ OK — 背景装飾用の空要素 -->
<div class="hero">
  <div class="hero__bg-circle" aria-hidden="true"></div>
  <h1>メインタイトル</h1>
</div>

<!-- ✅ OK — 見た目だけの区切り記号 -->
<nav aria-label="パンくずリスト">
  <a href="/">HOME</a>
  <span aria-hidden="true"> &gt; </span>
  <a href="/about">会社概要</a>
</nav>
対象aria-hidden="true"補足
テキストの隣にある装飾 SVG / アイコンつけるテキストが意味を担っているのでアイコンは不要
背景装飾用の空 div / spanつける読み上げに情報がない
パンくず等の区切り記号（> / |）つける読み上げると逆にノイズになる
「NEW」等の装飾バッジ（テキストで同内容が書かれている場合）つける見出し等で同じ情報が伝わるなら二重読みを防ぐ
意味のあるアイコン（テキストなし）つけないaria-label で機能を伝える（前述のアイコンボタン参照）
注意aria-hidden="true" をつけた要素の中にフォーカス可能な要素（a、button、input 等）を含めない。フォーカスは当たるが読み上げられない状態になり、操作不能になる。

装飾的な句読点・記号は CSS 擬似要素で出す

見出しやリストの装飾として句読点・記号（/、|、・、- 等）を入れる場合は、HTML ではなく CSS の ::before / ::after で出す。

装飾的なコンテンツは見た目の責務なので、HTML（構造）ではなく CSS（表現）で扱うのが正しい。HTML に書いて aria-hidden で隠す方法でも動くが、「意味のある要素を後から消す」ことになるため、最初から CSS で出す方がセマンティクス的に正しい。

<!-- ✅ OK — CSS 擬似要素で装飾の区切りを出す -->
<ul class="tag-list">
  <li>デザイン</li>
  <li>コーディング</li>
  <li>ディレクション</li>
</ul>
/* ✅ 区切り記号は CSS で出す → 読み上げに影響しない */
.tag-list {
  display: flex;
  list-style: none;
}
.tag-list li + li::before {
  content: "/";
  margin-inline: 0.5em;
}
<!-- ❌ NG — HTML に装飾の区切りを書いて aria-hidden で隠す -->
<ul class="tag-list">
  <li>デザイン</li>
  <li><span aria-hidden="true">/</span>コーディング</li>
  <li><span aria-hidden="true">/</span>ディレクション</li>
</ul>
方法評価理由
CSS 擬似要素（::before / ::after）推奨装飾は CSS の責務。読み上げに影響しない
span + aria-hidden="true"非推奨動くが、構造と表現が混在する

装飾目的で分割された要素の読み上げ対策

アニメーションやスタイリングのために1つの単語・数値を span で1文字ずつ分割するケースがある。この場合、スクリーンリーダーは span ごとに区切って読み上げてしまい、意味が伝わらない。

対策：分割された span を aria-hidden="true" で読み上げから除外し、外側の要素に aria-label で正しいテキストを設定する。

<!-- ❌ NG — span で分割すると「イチ」「ゼロ」「ゼロ」「パーセント」と読まれる -->
<div>
  <span>1</span><span>0</span><span>0</span><span>%</span>
</div>

<!-- ✅ OK — aria-label で正しい読み上げ、span は aria-hidden で除外 -->
<div aria-label="100パーセント">
  <span aria-hidden="true">1</span>
  <span aria-hidden="true">0</span>
  <span aria-hidden="true">0</span>
  <span aria-hidden="true">%</span>
</div>
<!-- ❌ NG — 「エ」「ー」「ビ」「ー」「シ」「ー」と1文字ずつ読まれる -->
<h2>
  <span>A</span><span>B</span><span>C</span>株式会社
</h2>

<!-- ✅ OK — h2 に aria-label、装飾部分は aria-hidden -->
<h2 aria-label="ABC株式会社">
  <span aria-hidden="true">A</span>
  <span aria-hidden="true">B</span>
  <span aria-hidden="true">C</span>
  株式会社
</h2>

このパターンが発生しやすい場面：

  • 数値のカウントアップアニメーション（100%、1,200件 等）
  • 1文字ずつ出現するテキストアニメーション
  • 文字間にスタイルを個別に当てるデザイン

</div>

属性用途

アコーディオン:

aria-expanded と aria-controls は 必ずセットで使う。aria-expanded だけでは「何が開閉するのか」がスクリーンリーダーに伝わらない。

<!-- ✅ OK — aria-expanded + aria-controls で開閉対象を紐づけている -->
<h3>
  <button aria-expanded="false" aria-controls="accordion-1" data-action="toggle-accordion">
    よくある質問 Q1
  </button>
</h3>
<div id="accordion-1" role="region" aria-labelledby="accordion-1-heading" hidden>
  <!-- 回答の内容 -->
</div>

<!-- ❌ NG — aria-expanded だけで aria-controls がない -->
<h3>
  <button aria-expanded="false" data-action="toggle-accordion">
    よくある質問 Q1
  </button>
</h3>
<div data-element="accordion-1" hidden>
  <!-- 開閉対象との関連付けがされていない -->
</div>
属性用途単体で使えるか
aria-expanded開閉状態を伝える。JS で true / false を切り替える不可。aria-controls とセットで使う
aria-controlsボタンが制御する要素の id を紐づけるaria-expanded と併用する
実装時の注意点 開閉対象の要素に id を設定し、ボタンの aria-controls にその id を指定する。JS で開閉を切り替えるとき、aria-expanded の true / false も必ず連動させる。この関連付けがないと「ボタンは開閉状態を持っているが、何を開閉するのかわからない」状態になる。

// ✅ フォーカストラップの基本実装
function trapFocus(modalElement) {
  const focusable = modalElement.querySelectorAll(
    'a[href], button:not([disabled]), input:not([disabled]), select:not([disabled]), textarea:not([disabled]), [tabindex]:not([tabindex="-1"])'
  );
  const first = focusable[0];
  const last = focusable[focusable.length - 1];

  modalElement.addEventListener('keydown', (e) => {
    if (e.key !== 'Tab') return;
    if (e.shiftKey && document.activeElement === first) {
      e.preventDefault();
      last.focus();
    } else if (!e.shiftKey && document.activeElement === last) {
      e.preventDefault();
      first.focus();
    }
  });
}
